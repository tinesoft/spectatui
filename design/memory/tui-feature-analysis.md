## Doc/Code Mismatches

- **README.md is accurate and up to date** for catalogs: its Features list and key-binding tables (`README.md:166`, `205-221`) correctly describe the Catalogs popup (`c` key, add/remove/refresh, tab across resource kinds) — this matches `catalogs.rs`/`main.rs` exactly. No mismatch here.
- **`specs/001-spectatui-dashboard-mvp/spec.md`'s "Out of Scope" note (line 214) is now stale in substance, not just framing.** It states catalog search/browse of not-yet-installed extensions, presets, and workflows is "intentionally out of scope" — but this capability is implemented today (see catalog item browse row above), just via a direct catalog-fetch mechanism rather than the CLI's own `search`/`list --available` subcommands. Recommend rewording to: catalog item discovery is implemented via a background catalog index, not via `specify <x> search`; only the literal CLI-subcommand path remains unused. Recommend also updating to clarify catalog *source* management (list/add/remove of catalog URLs) is separately and fully implemented (`22e80a7`).
- **`design/core/spectatui-archi-design.md` documents intent that code has since caught up to**: lines 144-151 describe extensions/presets managers as eventually supporting "search catalogs, view installed/available" items — the code now delivers this (via the direct-fetch path), even though `spec.md`'s Out of Scope note hasn't caught up.
- **No doc anywhere mentions `SelfCheck`/`SelfUpgrade`** even as a future item — the enum variants in `cli.rs:130-131` appear to be test scaffolding (arbitrarily chosen "some CliAction" for gating tests) rather than a deliberate, documented step toward implementing CLI self-check/upgrade.
- **No doc anywhere mentions `bundle`** — confirms this is a genuinely new upstream concept with zero design lineage in this repo.
- **Architecture note**: catalog item discovery bypasses the `specify` CLI subprocess entirely in favor of direct HTTP fetch of catalog JSON (`registry.rs`'s `reqwest` calls). This doesn't violate the "mutations go through the real CLI" principle (FR-020) since it's read-only discovery, not a mutation, but it does mean catalog-item metadata can drift from what `specify <x> search` would report if upstream's search logic ever diverges from the raw catalog JSON shape.

## Known Out-of-Scope Gaps (per spec.md)

1. **"Catalog search/browse of not-yet-installed extensions, presets, and workflows"** (`spec.md:214`) — **this text is now inaccurate and should be updated.** Catalog *source* management (list/add/remove of catalog URLs, all four kinds) was closed by `22e80a7`. Catalog *item* browse/search — the actual subject of this Out of Scope line — is **also implemented**, via the direct catalog-fetch mechanism described above, not via the CLI's `search`/`list --available` subcommands (those remain genuinely dead code in `cli.rs`, unused by any UI path). The only real remaining gap here is that the literal CLI subcommand path is unused — an internal implementation-path detail, not a user-facing capability gap. If Spectatui ever wants item discovery to go through `specify <x> search` instead of raw catalog JSON (e.g. to inherit upstream's own filtering/ranking, auth, or private-catalog support), that would be new work, but the current direct-fetch approach already satisfies the end-user need.
2. **"Self-check/self-upgrade of the underlying CLI tool"** (`spec.md:214`) — **still an open gap**, and here the CLI-invocation plumbing is already half-built: `CliAction::SelfCheck`/`SelfUpgrade` and their exact `specify self check`/`specify self upgrade` command lines exist in `cli.rs`, but with zero real UI wiring (only test-fixture usage in `app.rs`). Closing this gap needs a UI entry point (e.g., a command-palette action or a small "CLI" section in Settings) that constructs and confirms these two already-defined actions — no new upstream-CLI-shape work needed on Spectatui's side.
